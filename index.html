Thoughts on Progressive Enhancement

Progressive enhancement, simply put, is a mindset of web developement that focuses on the content of a webpage. That is to say, no matter what browser a user may be equipped with, a web developer builds their site with the intention that it is universally accessable. That is not to say that a web developer can't make use of newer features, but those features will simply act as the cherry on top to what is already easily accessable to all users of as many browsers as possible. The takaway from all this? Content is the focus, and content is what any user can absorb, no matter how the developer intended to present it. Looking at the web today, this has become a very important aspect. There are dozens of browsers out there, and the expectation to customize every page to every nuance of every browser us unrealistic. However, with progressive enhancement, at the very least, each browser can display the big picture of a web page. Additional Article

Feedback Writing Assignment

The first thing that become acutely aware to me when working on this project was the difficulty writing the css soon after starting. It took me three attempts to settle on an acceptable markup for the first draft. But I was suprised to notice a real progressive feeling of learning from each attempt. The final attempt was considerably more refined than the first, and I had a better understanding on how to make use of html classes and IDs. Doing individual excersices were relatively easy to comprehend, but the project is definitely an eye-opener in how to use our newly aquired toolkit. But that only came with considerable trial and error.

Working on the second draft, I'm noticing various obtimal changes to make to the css, such as removing fixed units of measurements in places where responsive units would be preferred. From the feedback I have received from peers, it mostly concerned minute changes to text content. For instance, one had confused a tab in the navigation bar for an about page labeled "Mission", as in "mission-statement." I've since renamed and clarified it as "Mission History". Another said captions would help them better understand the images displayed. I agreed since some of them on reflection do look hard to distinguish.

Additionally, the css has been restructured for easier navigation and comprehension. If I were to do the project over again, I would reconsider the way in which I went about initially writing the html, as when I was writing the css, I at times found myself having to create additional elements of html to be able to modify existing ones so that the css would affect them in an ideal way. When it came to moving images around the screen, I kept them contained within section tags paired with partner text elements, such that text on the left would have images on the right and vice versa. But I'm wondering if that was the best method for doing so since the images seem to be harshly affected elements on different viewports.

The only other feedback I was able to recieve was that the site didn't have much "pop". This is a relatively small site with mostly static text content, nonetheless I did my best to address that with the introduction of some javascript. Overall, the project was incredably beneficial to getting used to working with the different skills we've acquired this semester.
